# Vercel / Railway 是什么、怎么用？后端服务器与云服务对比

用通俗说法把「Vercel」「Railway」「后端服务器」「澳洲服务器 vs AWS」讲清楚，方便你后面选部署方案。

---

## 一、Vercel 是什么？怎么用？

### 1. Vercel 是什么

**Vercel** 是一家做「前端托管 + 开发体验」的公司，主要做两件事：

- **托管你打包好的前端**：你把 React 项目用 `npm run build` 打成静态文件（HTML、JS、CSS），Vercel 帮你放到全球的 CDN 上，任何人通过一个网址就能访问，而且**自动 HTTPS、自动全球加速**。
- **可选：跑一小段后端逻辑（Serverless 函数）**：比如一个「联系表单提交」接口，不用你买服务器，Vercel 在云端按请求临时跑一段代码，跑完就停。

可以简单理解成：**专门给「网站前端 + 轻量 API」用的托管平台**，你只管代码和 Git，不用管服务器、机房、SSL。

### 2. 和你项目的关系

- 你的 **前端**（React 页面、简历 PDF、会议论文 PDF 等）非常适合放在 Vercel 上：连上 GitHub，选好 `frontend` 目录和构建命令，每次 push 就自动重新部署。
- 你的 **联系表单** 目前是请求自己的 Node 后端；如果后端放在别处（比如 Railway），前端里配置一下后端地址即可；如果不想单独搞后端，也可以把「提交表单」这一块改成 Vercel 的一条 Serverless 接口（需要小改代码）。

### 3. 怎么用 Vercel（大步骤）

1. **注册**：打开 [vercel.com](https://vercel.com)，用 GitHub 账号登录。
2. **导入项目**：点「Add New Project」，选择你的 GitHub 仓库（例如 `feng's webpage` 对应的仓库）。
3. **配置前端**：
   - **Root Directory**：选 `frontend`（不要用整个仓库根目录）。
   - **Build Command**：`npm run build`。
   - **Output Directory**：`dist`。
   - 其他一般用默认即可。
4. **环境变量（若后端在别处）**：在 Project → Settings → Environment Variables 里加一条，例如：  
   `VITE_API_URL` = `https://你的后端地址`（没有协议尾部的斜杠即可）。
5. **部署**：点 Deploy，等几分钟，会给你一个 `xxx.vercel.app` 的地址，所有人就能访问你的前端了。之后每次往该仓库 push，Vercel 会自动重新构建、发布。

**总结**：Vercel = 托管前端（+ 可选小接口），连 GitHub、填构建命令就能用，不用自己买服务器。

## 二、Railway 是什么？怎么用？

### 1. Railway 是什么

**Railway** 是一个「应用托管平台」：你给它一份**会长期运行的程序**（比如一个 Node 服务器），它帮你在云端**一直跑着**，并给你一个对外的网址和 HTTPS。

和 Vercel 的对比可以这么记：

- **Vercel**：擅长「静态网站 + 按请求触发的短时函数」；你的 React 前端很适合放这里。
- **Railway**：擅长「一直运行的服务」；你的 Node 后端（`node server.js`）就是这种——需要 24 小时跑着，随时响应前端的 `/api/contact` 等请求。

所以常见做法是：**前端放 Vercel，后端放 Railway**，两个配合起来就是一个完整网站。

### 2. 和你项目的关系

- 你的 **backend**（Express 写的 `server.js`）在 Railway 上会像在本地一样运行：监听一个端口，处理 `GET /api/health`、`POST /api/contact`。
- Railway 会给你分配一个公网域名（如 `xxx.railway.app`），前端里的 `VITE_API_URL` 就填这个地址，表单就会提交到 Railway 上的后端。

### 3. 怎么用 Railway（大步骤）

1. **注册**：打开 [railway.app](https://railway.app)，用 GitHub 登录。
2. **新建项目**：New Project → 选「Deploy from GitHub repo」，选你的同一个仓库。
3. **指定后端目录**：在项目里会看到「从仓库部署」；需要把**根目录设为 `backend`**（或只部署 backend 这个目录，视 Railway 界面选项而定）。这样 Railway 只会用你后端的代码和 `package.json`。
4. **启动命令**：一般填 `npm start` 或 `node server.js`（和你本地一样）。Railway 会自动执行 `npm install` 再执行启动命令。
5. **环境变量**：在 Railway 项目里找到 Variables，可设：
   - `CORS_ORIGIN` = 你的前端地址（如 `https://xxx.vercel.app`），避免浏览器拦请求。
   - 其他你在代码里用 `process.env.xxx` 读的变量也可以在这里设。
6. **拿到公网地址**：部署成功后，Railway 会分配一个 URL；有的需要你在设置里给服务「生成一个 Public URL」。把这个 URL 填到前端的 `VITE_API_URL` 里（Vercel 环境变量里），再重新部署一次前端。

**总结**：Railway = 帮你「一直跑着 Node 后端」的平台，连 GitHub、选 backend 目录、设启动命令和环境变量即可，不用自己买服务器。

## 三、「后端服务器」到底指什么？几种常见形态

下面说的「后端」都指：**跑你写的服务端程序（如 Node/Express）的那台机器或那一层服务**。区别在于：这台「机器」是谁提供的、在哪里、你怎么用。

### 1. 你自己租的「一台机器」：VPS / 云主机

- **是什么**：你向某家厂商租**一整台虚拟机**（或物理机），有固定 IP、 root 权限，上面装什么系统、跑什么软件都是你说了算。
- **例子**：Vultr 上开一台「Sydney 机房」的 VPS；或 AWS 上开一台 EC2。
- **特点**：
  - **位置可选**：可以选澳洲机房，所以常说「澳洲后端服务器」往往指这种——服务器物理在澳洲，延迟对澳洲用户更友好。
  - **完全自控**：装 Node、Nginx、PM2、数据库，都自己来；也要自己管安全、备份、系统更新。
- **适合**：想完全掌控、要澳洲节点、或以后要跑数据库/复杂应用的人。

### 2. 大厂的「云服务」：AWS / 阿里云 / Azure 等

- **是什么**：除了「租一台虚拟机」以外，还提供一大堆**现成组件**：对象存储、数据库、消息队列、负载均衡、自动扩缩容等。你可以只用虚拟机（EC2），也可以把这些组件拼起来用。
- **和「澳洲后端服务器」的关系**：
  - **机房位置**：AWS 在悉尼有区域（ap-southeast-2），你在那里开 EC2，就是「在澳洲的服务器」；所以**地理位置**上，和你在 Vultr 选 Sydney 没有本质区别，都是「机器在澳洲」。
  - **差别**在于：AWS 还提供 RDS、S3、Lambda 等一大堆服务，你可以在同一账号下组合使用；而很多小厂 VPS 只给你「一台干净机器」。
- **优点**：功能多、生态成熟、大企业都在用、合规/认证多。
- **缺点**：计费复杂、学习曲线陡、对小项目容易「杀鸡用牛刀」。

### 3. 「别人帮你跑应用」的平台：Railway / Render / Heroku 等

- **是什么**：你不直接「租机器」，而是**把代码（和启动方式）交给平台**，平台负责找机器、装环境、跑你的进程、做重启和 HTTPS。你只关心「我的 Node 要跑起来、要有一个 URL」。
- **和「澳洲服务器」的对比**：
  - 这类平台**不一定**让你选「机房在澳洲」；很多是自动选一个区域（例如美东、欧洲），所以**延迟**上可能不如你专门在澳洲开一台 VPS。
  - 换来的是**不用管机器**：不装系统、不配 Nginx、不申请证书，适合「先把网站跑起来」。
- **适合**：个人项目、小团队、不想搞运维的人。

## 四、澳洲后端服务器 vs AWS 这类云服务：怎么比？

这里把「澳洲后端服务器」理解为：**机房在澳洲、主要用来跑你后端程序的那台机器（或那一组资源）**。和 AWS 的对比可以从「机房位置」和「服务形态」两方面看。

### 1. 机房位置（延迟、合规）

| 维度                  | 澳洲机房（如 Vultr Sydney、AWS ap-southeast-2） | 非澳洲机房（如美东、欧洲）         |
| --------------------- | ----------------------------------------------- | ---------------------------------- |
| **澳洲访客延迟**      | 低，数据在本地                                  | 高一些，数据要跨洋                 |
| **国内/其他地区访客** | 可能略慢（看具体线路）                          | 可能更快（若选离他们近的区）       |
| **合规/数据本地化**   | 若业务要求「数据留在澳洲」，选澳洲机房更合适    | 数据在境外，要看当地法律和公司政策 |

所以：**若你特别在意「澳洲用户体验」或「数据在澳洲」**，选**在澳洲有机房的**服务（无论是 Vultr、AWS Sydney 还是别的）更合适；**若主要面向全球或国内**，机房选离主要用户近的即可，不必强求澳洲。

### 2. 服务形态：VPS vs AWS 云

| 维度                   | 只租「澳洲 VPS」（如 Vultr Sydney）       | 用「AWS（可在悉尼开 EC2 + 其他服务）」                      |
| ---------------------- | ----------------------------------------- | ----------------------------------------------------------- |
| **你在澳洲的「后端」** | 就是那一台虚拟机，自己装 Node、Nginx、PM2 | 可以是 EC2（类似一台虚拟机），也可以配合 RDS、S3、Lambda 等 |
| **复杂度**             | 低：就一台机器，会 Linux 就能玩           | 高：产品多、计费细、要学一堆概念                            |
| **扩展性**             | 要更多性能就升配或再开几台，自己搞        | 可以用自动伸缩、负载均衡、托管数据库等                      |
| **成本**               | 通常月费固定、好算（如 5–10 美元/月）     | 按用量计费，小项目可能很便宜，用多了要小心账单              |
| **适合**               | 个人站、小项目、想练手运维                | 要高可用、要接一堆 AWS 组件的项目                           |

**简单结论**：

- **「澳洲后端服务器」**：强调的是**地理位置在澳洲**，可以是 VPS，也可以是 AWS 在悉尼的 EC2。
- **「AWS 这类云服务」**：强调的是**一整套云产品**（算力、存储、数据库、网络等），其中算力部分在悉尼用，就是「澳洲的云服务器」。
- **优劣**：同是澳洲机房时，VPS 更简单、成本更直观；AWS 更灵活、功能更多，但更复杂。对你现在的个人主页来说，**用 Vercel + Railway 先上线**，等真有需求（例如必须澳洲节点、或要接数据库）再考虑「租澳洲 VPS」或「用 AWS 悉尼」会更清晰。

## 五、小结：你现在可以怎么记

1. **Vercel**：托管前端（和可选的小接口），连 GitHub、填构建命令即可用；**不用**自己管服务器。
2. **Railway**：托管「一直跑着的」Node 后端，连 GitHub、选 backend、设启动命令和 CORS 等；**不用**自己买机器。
3. **后端服务器**：跑你服务端程序的那一层；可以是「你租的一台 VPS」、可以是「AWS 的一台 EC2」、也可以是「Railway 帮你跑的那一层」——形态不同，但都是「后端」。
4. **澳洲后端服务器**：指**机房在澳洲**的后端，延迟对澳洲用户友好；可以是澳洲 VPS，也可以是 AWS 悉尼区。
5. **AWS 等云服务**：一大套云产品；在悉尼用 EC2 就是「澳洲的云服务器」，优势是功能多、可扩展，劣势是复杂、计费要留心。

你先把「Vercel 管前端、Railway 管后端、不买服务器也能让人访问」这条线搞明白，再根据需要决定要不要上「澳洲 VPS」或「AWS 悉尼」会更容易选。
