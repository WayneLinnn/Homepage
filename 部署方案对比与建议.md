# 让所有人能访问你的网站：部署方案对比与建议

你的项目是 **React 前端 + Node.js 后端**（联系表单会请求后端 API）。下面几种方式都能让网站公网可访问，按「是否租服务器」「是否用 GitHub」等维度对比，并给出利弊与建议。

---

## 一、方案总览

| 方案                                      | 前端放哪       | 后端放哪            | 月成本          | 难度 | 适合场景                                     |
| ----------------------------------------- | -------------- | ------------------- | --------------- | ---- | -------------------------------------------- |
| **A. GitHub Pages + 后端另放**            | GitHub Pages   | Railway / Vercel 等 | 0（免费额度内） | 中   | 想用 GitHub 展示、接受「前后端分开部署」     |
| **B. 租一台服务器 (VPS)**                 | 同服务器 Nginx | 同服务器 Node       | 约 5–10 美元    | 高   | 想完全自控、练运维、或要澳洲机房             |
| **C. Vercel + Railway**                   | Vercel         | Railway             | 0（免费额度内） | 低   | 不想管服务器，快速上线、联系表单要能用       |
| **D. 全 Vercel（前端 + Serverless API）** | Vercel         | Vercel 云函数       | 0               | 低   | 能接受把联系接口改成一条云函数，一个平台搞定 |

## 二、各方案说明与利弊

### 1. 放到 GitHub 上展示（GitHub Pages）

**能做什么**

- 把前端的 **构建结果**（`npm run build` 后的 `dist`）部署到 **GitHub Pages**，得到 `https://你的用户名.github.io/仓库名/` 这样的地址，所有人都能访问你的**前端页面**。

**限制**

- GitHub Pages **只提供静态文件**（HTML/CSS/JS），**不能跑 Node.js**。
- 你的**联系表单**依赖后端 `POST /api/contact`，所以：
  - **仅用 GitHub Pages**：前端可以展示，但提交表单会失败（没有后端）。
  - **若要坚持「放 GitHub」**：前端用 GitHub Pages，**后端必须单独部署**到别处（如 Railway、Vercel Serverless），并在前端配置 `VITE_API_URL` 指向该后端。

**优点**

- 免费、和 GitHub 仓库天然结合、适合做作品集展示。
- 前端部署流程简单（Git push → 自动构建/发布）。

**缺点**

- 后端要另找地方，不是「全部在 GitHub」就完事。
- 若后端免费额度用尽或服务不稳定，表单会受影响。

**结论**：**可以**用 GitHub Pages 展示前端，但若你要联系表单可用，必须再选一个后端方案（见下面 C 或 D）。

---

### 2. 租一台服务器 (VPS)

**做法**

- 在 Vultr、DigitalOcean、AWS 等租一台虚拟机（可选澳洲机房），自己装 Node、Nginx，把前端静态文件交给 Nginx，用 PM2 跑后端 `node server.js`。

**优点**

- **完全自控**：数据、日志、域名、防火墙都在自己手里。
- 可选**澳洲节点**，本地/澳洲访客延迟低。
- 以后要加数据库、定时任务、别的服务都方便。

**缺点**

- **要自己维护**：系统更新、安全、备份、SSL 证书（如 Let’s Encrypt）都要管。
- **有月费**：约 5–10 美元/月起。
- **上手成本高**：要会一点 Linux、Nginx、PM2。

**结论**：适合愿意花时间学运维、或明确需要「自己的服务器」的人；对「先尽快让人访问到」不是最优。

---

### 3. Vercel（前端）+ Railway（后端）

**做法**

- **前端**：在 Vercel 连 GitHub 仓库，根目录选 `frontend`，构建命令 `npm run build`，输出 `dist`，自动 HTTPS。
- **后端**：在 Railway 连同一仓库，根目录选 `backend`，启动命令 `node server.js`，Railway 会给你一个公网 URL。
- 在前端环境变量里设 `VITE_API_URL=https://你的 Railway 地址`，后端 CORS 里允许 Vercel 的域名。

**优点**

- **几乎零运维**：不用买服务器、不用配 Nginx。
- **免费额度**对个人主页通常够用（Vercel 免费，Railway 有一定免费额度）。
- **HTTPS、CDN** 都自带，全球访问速度不错。

**缺点**

- 前后端在两个平台，要分别看日志、配置环境变量。
- Railway 免费额度用完后要付费或换平台。

**结论**：**最推荐**的折中方案：不租服务器，又能保证前端 + 联系表单都可用，配置一次即可。

---

### 4. 全 Vercel（前端 + Serverless 联系接口）

**做法**

- 前端照常部署到 Vercel。
- 把现在的 `POST /api/contact` 改成 **Vercel Serverless Function**（例如 `api/contact.js`），不再单独跑一个 Node 服务器。
- 表单提交到 `https://你的域名/api/contact`，由 Vercel 云函数处理（写日志或发邮件等）。

**优点**

- **一个平台搞定**，不用管 Railway。
- 免费额度对个人站足够，HTTPS、全球分发自带。

**缺点**

- 要**改一点代码**：把 Express 的那段联系逻辑迁到 Vercel 的 API 路由里（逻辑很简单，主要是适配入参/出参）。

**结论**：若你愿意做一次小改造，这是**成本最低、步骤最少**的「全站上线」方案。

---

## 三、按你需求选哪个

- **「我就想用 GitHub 展示」**  
  → 前端用 **GitHub Pages**，后端用 **Railway** 或 **Vercel Serverless** 二选一；否则表单用不了。

- **「不想租服务器、也不想折腾」**  
  → **Vercel + Railway**（当前项目几乎不用改代码），或 **全 Vercel**（把 contact 改成一条 API 路由）。

- **「我要完全掌控、或必须用澳洲服务器」**  
  → **租 VPS**（如 Vultr Sydney），自己部署前后端。

- **「先让人能访问，表单暂时可以不要」**  
  → 只部署前端到 **GitHub Pages** 或 **Vercel**，联系页可以只留邮箱链接，之后再加后端或 Serverless。

---

## 四、建议顺序（不租服务器时）

1. **优先试：Vercel（前端）+ Railway（后端）**
   - 文档里已有步骤，改环境变量即可，前后端都不大改。
   - 做完后把「前端网址」和「full-stack-ai-developer-feng-lin-2026.pdf / 会议论文 PDF」等链接检查一遍即可。

2. **若希望一个平台、少维护**
   - 再考虑把联系接口迁到 **Vercel 的 `/api/contact`**，去掉对 Railway 的依赖。

3. **若以后要澳洲机房或完全自控**
   - 再买 VPS，把现在在本地跑的那套（Nginx + PM2）搬到服务器上即可。
