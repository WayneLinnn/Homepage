## 二、代码与实现参考（看技术栈与结构）

### 1. 现代技术栈与性能

| 来源 / 项目                                                    | 技术要点                                                                            | 对你这边的启发                                                                                      |
| -------------------------------------------------------------- | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **Astro + React 组合示例**（如 sdburt.com 的 Astro portfolio） | Astro 默认零 JS，需要交互的部分用 React；Lighthouse 满分；设计系统 + 可访问性优先。 | 即使你当前用纯 React，也可以想：哪些区块可以“静态”、哪些需要交互；图片优化、语义化 HTML、键盘导航。 |
| **Next.js + Framer Motion 类作品集**                           | TypeScript、Framer Motion 做页面过渡与微动效、Tailwind 或设计 token、MDX 写内容。   | 你已用 React：可考虑加 Framer Motion 做“进入/离开”动画；用环境变量区分开发/生产 API。               |
| **GitHub: react-portfolio-2024、ReactJS-portfolio-website 等** | 纯 React 作品集：组件划分、路由、响应式、简历 PDF、项目列表。                       | 直接看目录结构（components / pages / assets）、如何拆 Header、Hero、ProjectCard、Contact 等。       |

### 2. 可重点看的代码习惯

- **组件划分**：按区块拆（Header、Hero、About、Projects、Contact），而不是一个巨大 App。
- **样式方式**：要么 CSS Modules / 单一全局 + BEM，要么 Tailwind；保持一种风格。
- **动效**：用 CSS `transition` / `animation` 或 Framer Motion，并做 `prefers-reduced-motion` 降级。
- **可访问性**：语义化标签（`<header>`、`<main>`、`<section>`、`<nav>`）、按钮/链接可键盘聚焦、表单带 `label`。
- **性能**：图片用合适尺寸或 `srcset`；若以后加很多动效，注意 `will-change` 或只对必要元素做动画。

## 三、设计趋势小结（便于你“有依据”地做决策）

| 趋势                 | 简要说明                                                  | 你可如何用                                                                      |
| -------------------- | --------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **极简 + 留白**      | 少装饰、多留白、信息分层清晰。                            | 你当前深色 + 红色强调已经偏极简，可保持；重点把“关于 / 项目 / 联系”分区更清晰。 |
| **大标题与字体层级** | 首屏用大号字体做姓名或一句 Slogan；正文、小标题层级分明。 | Hero 区姓名或一句话加大、加字重；其余用 1～2 级标题 + 正文字号。                |
| **深色模式**         | 深色背景 + 浅色字，护眼且显专业。                         | 你已用深色，可保持；若以后做亮色版，可做主题切换。                              |
| **适度动效**         | 滚动出现、悬停反馈、页面切换过渡，但不过度。              | 先做 1～2 处：如导航 hover、卡片 hover、或“联系”按钮的过渡。                    |
| **响应式**           | 手机端单列、触控友好、字体不缩小到难以阅读。              | 你已有基础响应式；可检查 375px 宽度下导航、表单、按钮是否好用。                 |
| **可访问性**         | 键盘可操作、对比度够、屏幕阅读器友好。                    | 给表单加 `<label>`、链接/按钮可 Tab 聚焦、颜色不唯一传达信息。                  |

## 四、对接下来的开发可以怎么“思考”

1. **先定 1～2 个参考站**  
   在 Awwwards 或 developer-portfolios 里选 1～2 个你喜欢的，只借鉴：**首屏布局、项目卡片样式、整体配色**，不必照抄。

2. **信息架构**  
   你已有：首页、关于、项目、简历、联系。可再想：
   - 首页要不要一句“一句话介绍”或 Slogan？
   - 项目是“标题 + 简短描述 + 链接”就够，还是要“问题-做法-结果”小案例？
   - 简历是内嵌 PDF 预览还是只放下载链接？（你当前是链接，已足够。）

3. **视觉进阶（可选、逐步加）**
   - 字体：选一款无衬线做主字体（如 Inter、DM Sans），必要时加一款衬线做标题。
   - 动效：先做导航与卡片 hover、再考虑滚动进入动画（如 Framer Motion 或 CSS）。
   - 可访问性：检查焦点样式、对比度、表单标签。

4. **代码结构**
   - 保持 `components`（Layout、Button、ProjectCard 等）与 `pages` 分离。
   - 若加动效库，统一在一个文件夹（如 `components/animations` 或工具函数）。
   - API 调用继续用环境变量 `VITE_API_URL`，便于本地 3000 与线上后端切换。
