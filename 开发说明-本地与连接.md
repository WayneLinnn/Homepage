# 开发说明：本地端口、前后端连接、澳大利亚服务器

---

## 1. localhost:3000 是哪里用到的？前端怎么连后端？

### 重要区分：本地开发 vs 部署到服务器

| 环境             | 前端怎么“传输”                                                                                                                                                   | 后端怎么“传输”                                                                                                |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| **本地开发**     | 用 **5173**：你运行 `npm run dev`，Vite 开发服务器在 5173 端口，浏览器打开 `http://localhost:5173` 看页面。                                                      | 用 **3000**：你运行 `node server.js`，后端在 3000 端口，提供 `/api/health`、`/api/contact` 等接口。           |
| **部署到服务器** | **不用 5173**。前端先 `npm run build` 打成静态文件（HTML/JS/CSS），由 Nginx 或 Vercel 等用 **80/443** 提供访问，用户看到的网址是 `https://你的域名`，没有 5173。 | 用 **3000**（或平台分配的端口）：`server.js` 在服务器上跑，监听 3000（或 `process.env.PORT`），对外提供 API。 |

所以：**5173 只在本地开发时用**，是 Vite 的开发服务器；**部署到服务器后，前端是静态文件，通常用 80/443，不是 5173**。**3000 是后端用的**，本地和服务器上都是（服务器上也可能被平台改成别的端口，由环境变量 `PORT` 决定）。

### 谁在用什么端口？（本地开发时）

| 端口               | 谁在用                      | 作用                                                                       |
| ------------------ | --------------------------- | -------------------------------------------------------------------------- |
| **localhost:5173** | **前端**（Vite 开发服务器） | 你在浏览器里打开的页面：`http://localhost:5173`，所有 React 页面都在这里。 |
| **localhost:3000** | **后端**（Node.js Express） | 提供 API：健康检查、联系表单提交等，**不直接**在浏览器地址栏打开。         |

也就是说：**3000 是后端 API 服务器用的端口**，不是前端页面地址。

### 前端怎么“连”到后端？

本地开发时，流程是这样的：

```
你在浏览器打开: http://localhost:5173  （前端）
        ↓
你点击「联系」页、填写表单、点「发送」
        ↓
前端代码发请求: fetch("/api/contact", { method: "POST", ... })
        ↓
这个请求的 URL 浏览器会认为是: http://localhost:5173/api/contact
        ↓
Vite 开发服务器看到路径是 /api 开头 → 按配置把请求转给后端
        ↓
实际请求被转发到: http://localhost:3000/api/contact  （后端）
        ↓
后端 Express 处理 POST /api/contact，写入 data/contact.json，返回 JSON
        ↓
Vite 把后端的响应再转回给浏览器
        ↓
前端收到响应，显示「已提交」或错误信息
```

所以：**前端不直接写 localhost:3000**，而是写相对路径 `/api/contact`；**由 Vite 的 proxy 把 /api 的请求转到 localhost:3000**。这样做的原因：

- 避免跨域（CORS）问题：浏览器里看到的请求还是同源（都是 5173）。
- 换环境时只需改一处：上线后把前端的 `VITE_API_URL` 设成真实后端地址即可。

### 在代码里具体是哪里？

1. **后端用 3000 端口**  
   文件：`backend/server.js`
   - `const PORT = process.env.PORT || 3000;`
   - `app.listen(PORT, ...)`  
     所以后端跑起来后监听的是 `http://localhost:3000`。

2. **前端“连后端”的配置**  
   文件：`frontend/vite.config.js`
   - `proxy: { "/api": { target: "http://localhost:3000" } }`  
     意思：所有路径以 `/api` 开头的请求，在开发时被 Vite 转发到 `http://localhost:3000`。

3. **前端发请求的代码**  
   文件：`frontend/src/pages/Contact.jsx`
   - `const API_BASE = import.meta.env.VITE_API_URL || "";`
   - `fetch(\`${API_BASE}/api/contact\`, ...)`
   - 本地没设 `VITE_API_URL` 时，`API_BASE` 是空字符串，所以请求的是 `/api/contact`（相对当前域名 5173），再由 Vite 转发到 3000。
   - 部署后：在 Vercel 等地方设置 `VITE_API_URL=https://你的后端域名`，请求就会发到澳大利亚（或别的）服务器上的后端。

## 2. 想把个人主页放在澳大利亚，怎么租服务器？

“澳大利亚的服务器”一般指：**机房在澳大利亚的 VPS/云主机**，这样访客（尤其是澳洲本地）访问延迟更低。

### 常见做法

| 方式                           | 说明                                                                | 适合                   |
| ------------------------------ | ------------------------------------------------------------------- | ---------------------- |
| **VPS（虚拟专用服务器）**      | 租一台在澳洲机房的虚拟机，自己装 Node、Nginx，部署前后端。          | 想完全自己控、学运维   |
| **PaaS（如 Railway、Render）** | 选区域时选 Sydney/Australia（若提供），或接受就近区域（如新加坡）。 | 想少管服务器、快速上线 |
| **云厂商澳洲区**               | 在 AWS / 阿里云等选“悉尼”区域开一台 EC2/ECS。                       | 已有云账号、要稳定可控 |

### 有澳大利亚机房的常见服务商

- **Vultr**：有 Sydney 机房，按小时计费，约 5–6 美元/月起。
  - 官网：https://www.vultr.com
  - 注册后 Deploy New Server → 选 Sydney。
- **DigitalOcean**：有 Sydney 数据中心。
  - 官网：https://www.digitalocean.com
  - Create Droplet → 选 Region: Sydney。
- **AWS**：区域选 **ap-southeast-2（Sydney）**，开 EC2。
- **Linode（Akamai）**：有 Sydney 机房。
- **Oracle Cloud**：有 Australia East (Sydney)，有免费档可薅。

### 租用步骤（以 Vultr 为例）

1. 注册账号，充值（如 10 美元）。
2. Deploy New Server → 选 **Sydney** → 选系统（Ubuntu 22.04）→ 选套餐（如 5 USD/月）。
3. 创建后拿到 **IP 地址** 和 **SSH 密码/密钥**。
4. 用 SSH 连上去：
   - 安装 Node.js、Nginx、PM2。
   - 把后端代码放上去，用 PM2 跑 `node server.js`。
   - 把前端的 `npm run build` 产物放到 Nginx 静态目录，或 Nginx 反代前端。
   - 配置域名（可选）和 HTTPS（如 Let’s Encrypt）。

如果你希望“尽量少管服务器”，可以：

- 前端继续用 **Vercel**（免费、自动 HTTPS）。
- 后端用 **Railway** 或 **Render**，选离澳洲近的区域（若没有悉尼就选 Singapore 等），这样也算“靠近澳大利亚”的部署。

## 3. 开发时想搞懂代码：关键概念解释

下面按「请求从页面到后端再回来」的顺序，把用到的概念简单说明。

### 前端（React）

- **组件**：页面上一块可复用的 UI + 逻辑，例如 `Contact.jsx` 就是“联系”这一页。
- **状态（useState）**：组件内部会变化的数据，例如表单里填的 name、email、message，以及“正在发送 / 成功 / 失败”。
- **事件**：用户操作触发的函数，例如 `onChange`（输入框内容变化）、`onSubmit`（表单提交）。
- **fetch**：浏览器提供的发 HTTP 请求的 API；这里用来把表单数据 POST 到后端的 `/api/contact`。
- **环境变量（VITE_API_URL）**：构建时注入的配置，用来区分“开发时用代理”和“上线时用真实后端地址”。

### 后端（Node.js + Express）

- **Express**：基于 Node 的 Web 框架，用来定义“收到 GET /api/health 时返回什么”“收到 POST /api/contact 时做什么”。
- **中间件**：按顺序执行的一串函数，例如 `cors()` 先处理跨域，`express.json()` 再把请求体解析成 JSON，然后才到你的 `app.post("/api/contact", ...)`。
- **CORS**：浏览器的安全策略；后端要显式允许“哪个前端域名可以调我”，这里开发时用 `*`，上线时改成你的前端域名。
- **rate-limit**：限制同一 IP 在短时间内调用次数，防止联系表单被刷。
- **req.body**：POST 请求里前端 `JSON.stringify(form)` 发过来的数据，被 `express.json()` 解析成对象，所以能解构出 `name, email, message`。

### 本地联调小结

- 你先在终端 1 运行 `cd backend && npm start`，后端在 **localhost:3000**。
- 再在终端 2 运行 `cd frontend && npm run dev`，前端在 **localhost:5173**。
- 浏览器只访问 **http://localhost:5173**；所有对 `/api/*` 的请求由 Vite 转发到 **localhost:3000**，这就是“前端连后端”的完整链路。
